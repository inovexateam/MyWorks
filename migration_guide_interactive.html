git clone https://your-org/your-aspnet-app.git cd your-aspnet-app code . # open in VS Code # copy .github/ folder from this solution into root cp -r /path/to/this-solution/.github ./ git add .github/ git commit -m "chore: add migration tooling"
That .github/ folder IS the entire tool. No extension to install beyond Claude Code.

One command scans your entire codebase

Run this in VS Code terminal. It builds CODEBASE-MAP.md automatically — listing every file, its complexity, and what needs migrating. This is the only time you wait for a full scan.

dotnet run --project .github/tools/Scanner -- --solution ./YourApp.sln

No scanner tool yet? Ask Claude Code directly:

@claude scan this solution and populate .github/memory/CODEBASE-MAP.md Include every .cs, .aspx, .ascx, .asmx file with LOC count and complexity. Flag: LDAP connections, ADO.NET, Apigee calls, Ping login, Crystal Reports.
What gets auto-detected and flagged
  • auto All System.Web.* dependencies per file
  • auto LDAP / DirectoryServices calls → flag for manual review
  • auto ADO.NET SqlConnection → migration path: Dapper or EF Core
  • auto HttpClient / RestSharp → Apigee calls flagged as external
  • auto FormsAuthentication → Ping/OIDC replacement needed
  • manual Crystal Reports (.rpt) → needs stakeholder decision
  • manual COM Interop / GAC → assess per component
Manual intervention #1 — after the scan, review the BLOCK entries. These are components with no automatic migration path. Decisions (Crystal Reports replacement, Apigee contract changes, Ping OIDC config) must be made by humans before those files can be migrated.

Creating the new .NET 8 project — single command

The new Core project is created alongside the old one. They live in the same git repo during migration. You do NOT throw away the old app until migration is 100% verified.

@claude create the new .NET 8 solution structure alongside this existing app. Use the project layout from CODEBASE-MAP.md. Create: YourApp.Core.sln with matching projects. Configure nuget.config to use Artifactory only.

Claude Code creates this structure:

your-repo/ ├── src-framework/ ← your EXISTING app (untouched) │ ├── WebApp/ │ ├── BC/, BPC/, DAC/ ... │ └── src-core/ ← NEW .NET 8 app (built here) ├── WebApp.Core/ ← Razor Pages replacing .aspx ├── BC.Core/ ← same business logic, no System.Web ├── BPC.Core/ ├── DAC.Core/ ← EF Core replacing EF6/ADO.NET ├── SAC.Core/ ← Apigee/external calls └── YourApp.Core.sln └── nuget.config ← Artifactory only, no nuget.org
Manual intervention #2 — fill in your Artifactory URL in nuget.config and add ARTIFACTORY_USER / ARTIFACTORY_TOKEN to your environment or .env file (never commit these).

Migration order — always leaf-first

The map enforces this order automatically. Each project only starts after its dependencies are ✅ DONE in the map.

Correct order
1. Utilities (no deps)
2. DAC (data access)
3. BC (uses DAC)
4. SAC (uses BC)
5. BPC (uses BC+SAC)
6. WebApp last
Single command per project
@claude run migration-bundle for project: Utilities using map: .github/memory/ CODEBASE-MAP.md

Claude reads the map, skips ✅ files, migrates ⏳ files, marks 🚧 blockers, updates map when done.

Manual intervention #3 — after each project completes, run dotnet build src-core/ and confirm 0 errors before moving to next project. Never migrate two projects simultaneously.

Your specific patterns — migration paths

Every pattern in your app has a defined route. No guessing needed.

LDAP / DirectoryServices→ Microsoft.Extensions.DirectoryServices or Novell.Directory.Ldap.NETStandard
ADO.NET SqlConnection→ Dapper (minimal change) or EF Core (full ORM)
Apigee external calls→ IHttpClientFactory + typed clients, API key in Key Vault
Ping / SAML login→ OpenID Connect middleware (AddOpenIdConnect), Ping as IdP
FormsAuthentication→ Cookie auth or ASP.NET Core Identity
.aspx pages→ Razor Pages (agent-ui-adapter handles all controls)
.ascx controls→ Partial Views or View Components
.asmx SOAP→ Minimal API or CoreWCF (agent-soap-wcf-migrator)
web.config→ appsettings.json + env vars + Key Vault
Crystal Reports🚧 needs stakeholder decision — 4 options in map
Manual intervention #4 — Ping/OIDC config needs your org's actual client ID, client secret, and discovery URL. These come from your identity team, not the code. Add them to Key Vault, never to appsettings.json.
